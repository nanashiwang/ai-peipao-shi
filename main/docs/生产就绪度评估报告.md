## 1. 系统概览

- **形态**：基于企业微信的教培机构「陪跑师效率系统」SaaS，服务未成年学生与家长的沟通场景。

**工程正面信号**（说明团队有正确直觉，不是从零）：`claim_lock` 领取锁用 `SKIP LOCKED` + 条件更新，并发设计扎实；`scenario`/`importer`/`send_log_classifier` 纯函数分层得当；`runtime_config` 有生产配置守卫；真实发送有「发送前基线 + 发送后回读 diff」双向确认与幂等去重；关键路径（鉴权/发送/数据隔离）都有测试。

## 2. 分维度评估

### 2.1 后端架构与代码质量（5.0/10）

**核心问题：`main.py` 是 5451 行、106 路由、258 函数的「上帝文件」，零 APIRouter 分层；仪表盘类接口存在系统性 N+1。**

- **[P0] 仪表盘 N+1，随客户数据量线性劣化直至超时**：`build_admin_service_quality_dashboard`（`main.py:3181-3259`）外层循环每个家庭调 `infer_family_service_stage`（内含 6 查询，`2949-2993`）+ 3 个 count + 一次 `SendLog.all()`，且 campus 循环里**又重复调一遍**。单请求 SQL 数 ≈ 家庭数 ×（10~12）。500 家庭 → 单次首页 5000+ 次 DB 往返。同类：`build_workbench_todos`、`build_today_priorities`、`list_families`。
- **[P0] 57 处 `commit()`、0 处 `rollback()`，无全局异常处理**：`get_db`（`db.py:22-27`）异常路径不回滚；长事务（`sync_conversation_payload` `3716-3884`、`claim_tasks` `5051-5139`）中途异常时「部分写入是否落库」语义脆弱。
- **[P1] 500 响应回吐原始异常文本**：`main.py:4020` `f"...失败：{exc}"` 把内部细节暴露给调用方。
- **[P1] 巨型编排函数**：`sync_conversation_payload` 一个函数扛「落消息 + 自动回复 + 建任务 + 批量生成 + 审计」5 类职责，失败被静默降级为 `auto_reply_note` 字符串。
- **[P1] 批量生成无预算控制**：`/api/generate/all`（`3505-3512`）每家庭串行 4 次 Ark 调用（每次 timeout 60s），N 家庭最坏 N×4×60s，必然网关超时。
- **[P2] 启动即裸跑 DDL**：`db.py:45-128` 每次启动 `ALTER TABLE ADD COLUMN` 并给布尔低基数列无脑建索引；`@app.on_event("startup")` 已废弃。

**投产前必修 Top 5**：① 重写仪表盘/列表 N+1 为批量聚合；② `get_db` 加 rollback + 全局异常处理器；③ 异常文本不外泄 + 用 Alembic 取代启动期 DDL；④ 批量 Ark 接口改后台任务 + 上限；⑤ 按领域拆 6–8 个 APIRouter。

### 2.2 安全（2.5/10）—— 最高优先级维度

**核心问题：系统按「本地试点优先、鉴权默认关闭」设计，导致默认/pilot 配置下处理未成年人数据的后端几乎完全暴露。**

- **[P0] 默认配置鉴权关闭**：`.env.example:14` 显式 `ADMIN_AUTH_REQUIRED=false`，照抄即裸奔（注：不设该变量时 pilot/production 会自动开启鉴权，`admin_auth.py:56-63`——但模板给的是显式 false）。中间件 `main.py:158` 未开鉴权时直接放行，全部家庭/聊天/发送接口无凭证可读写。
- **[P0] 公开端点签发管理员 token**：`/api/test-chat/login`（`main.py:2267-2273`，`PUBLIC_PATHS` 白名单内）对任意 `role in ADMIN_ROLES` 账号铸造 `admin_token`，测试口子成了正式发证机。
- **[P0] RPA 端点完全免鉴权**：`is_device_or_rpa_path` 对 `/api/rpa/` 一律豁免（`admin_auth.py:214-225`），`/api/rpa/conversations/resolve` 可匿名按家长昵称枚举家庭数据，`sync` 可匿名注入伪造聊天记录污染 AI 与发送链路。
- **[P0] 真实密钥明文落盘**：`config/ark.json`（真实 DashScope key）、`config/admin_secret.txt`、`rpa/config.json`（明文 device_token），且 `docker-compose.yml:17-21` 整个 `./config` 读写挂载进容器。**需立即轮换**。
- **[P0] 默认弱口令绕过校验**：`assert_runtime_config_safe` 仅对 `production/prod` 生效，compose 默认 `APP_ENV=pilot` 绕过全部校验，弱 DB 口令 `coach/coach` + 占位 secret 可预测。
- **[P1] send-artifacts 截图端点无鉴权 + 文件名可枚举**：`main.py:4647-4650`，文件名 `task_<自增id>_<时间戳>.png`，遍历即可拉取所有含未成年人聊天的截图（IDOR）。
- **[P1] 脱敏形同虚设**：`should_redact_for_request`（`main.py:560-566`）仅 `role==readonly` 才脱敏，coach/admin/匿名全部返回明文手机号、姓名、聊天原文。
- **[P1] CORS 全开** `allow_origins=["*"]`（`main.py:136-142`）；**无暴力破解防护**（仅按 IP 滑窗限流，重启清零，可伪造 X-Forwarded-For 绕过）。
- **[P1] 依赖版本落后存在已知漏洞风险**：`python-multipart==0.0.20`、`cryptography==43.0.3` 均落后多个安全版本（具体 CVE 编号请以 `pip-audit` 复核为准，不要凭编号下结论）。
- **[P2] RPA 被控端无签名信任链**（默认 unsigned，`device_token` 明文分发可冒充）；**明文 HTTP 传输**管理 token 与 PII。

**投产前必修 Top 5**：① 强制默认鉴权 + 拒绝弱口令/占位密钥启动（覆盖 pilot）；② 生产禁用 `/api/test-chat/*`，切断公开发证链；③ `/api/rpa/*` 强制 `require_device`；④ 立即轮换已泄露密钥，改环境注入禁明文落盘；⑤ send-artifacts 加鉴权 + 作用域 + 不可枚举文件名，脱敏改默认生效。

### 2.3 前端（6.5/10）

**核心问题：3432 行单文件手工渲染，XSS 防线 100% 依赖人工调 `esc()`，token 存 localStorage。**

- **[P0] XSS 无系统性兜底**：`innerHTML` 拼接 120+ 处全靠手写 `esc()`；且 `esc()`（`app.js:224`）不过滤 `javascript:` 协议，`app.js:830`/`2112` 截图路径直接进 `href`，被污染成 `javascript:...` 点击即执行（真实可利用点）。无 DOMPurify、无 CSP。
- **[P0] admin_token 存 localStorage**（`app.js:29/72/105`），XSS 可直接盗超管 token → 真实发送。这是「真实发送 + 账号体系」系统的最高危组合。
- **[P1] 单文件无模块化，45+ 函数挂全局**（inline `onclick` 依赖）；**每次写操作 `refreshAll()` 全量重渲**（`app.js:2354` 发 20 个请求 + 重绘 26 个面板，取消一个任务 = 全站重绘）；**批量操作串行请求瀑布**；**fetch 无超时无重试 + 错误原文直接 toast 给用户**（`app.js:74-75`，境外网络下可能永久 pending）。
- **[P2] 无构建**（无 bundle/min/懒加载/缓存指纹）；KPI 满意度评分在前端硬编码猜测（`app.js:897-908`）。

**工程建议**：**强烈建议立即引入 Vite（低风险高收益）**——可先把现有 `app.js` 原样打包拿到 min + hash + gzip，再渐进拆 ES Modules；一并接 ESLint（`no-unsanitized` 规则强制检测 innerHTML）。**框架不建议推倒重写**，中期可用 Preact+htm 或 Lit 按面板渐进组件化。TypeScript 先 `checkJs + JSDoc` 渐进接入。

### 2.4 数据治理（4.0/10）

**核心问题：投产用 PostgreSQL，却无迁移工具、备份对生产库失效、敏感截图无限留存。**

- **[P0] 无迁移工具**：`db.py:31-35` 裸 `create_all` + `45-128` 手写 `ALTER TABLE`，无 Alembic。`create_all` 永不改已存在表，投产后任何字段改名/类型变更/加约束都无迁移路径。
- **[P0] 备份对生产 PostgreSQL 完全失效**：`backup_service.py:18-27` 遇非 `sqlite:///` 直接 `raise ValueError`，而生产是 PG，`POST /api/ops/backups` 必然抛错 = **零备份**。看板却只报 warn，给虚假安全感。
- **[P0] 敏感聊天截图无限留存**：`retention_service` 清理正则只匹配 `task_\d+_...png`，**完全覆盖不到 `rpa/debug_*.png`**（114 张含真实未成年人聊天界面），无过期删除，永久明文。
- **[P1] `raw_messages.external_id` 无 DB 唯一约束**（`models.py:72` 仅 index），去重全靠应用层，并发/重复拉取会产生重复记录污染 AI 分析。
- **[P1] 全表无软删除**（物理删除不可逆）；**数据保留无定时调度**（`prune_retention` 只能手动触发，无 APScheduler/cron，保留策略形同虚设）；**外键仅 1 处**，大量 `family_id` 是裸字符串无 FK，删除家庭后孤儿记录无约束清理。
- **[P2] 连接池全默认**（无 `pool_pre_ping`/`pool_recycle`，长连接遇 PG 断连抛 stale）；时间戳全用 naive `utcnow`。

**投产前必修 Top 5**：① 引入 Alembic 冻结 baseline，生产禁用 create_all；② 实现 pg_dump 备份 + restore 演练 + 定时调度；③ 物理清除截图 + 生产关 debug + 纳入 retention；④ 改默认密钥拒绝启动；⑤ `raw_messages(family_id, external_id)` 加唯一约束改 upsert。

### 2.5 RPA 子系统（5.5/10）

**核心问题：防呆与发送闭环做得扎实，但「模拟企微 UI + 截图/OCR 定位」的地基决定了它无法独立支撑生产 SLA。**

- **[P0] result 上传截图无限堆积**：`wecom_sender.py:364-377` 每次真实发送落一张全屏 PNG，上传后**从不清理**（`_cleanup_debug_image` 未覆盖 result 截图）。被控端长期运行必然写满磁盘，磁盘满 → 全链路失败 → 阻塞发送队列。
- **[P0] device_token 明文存储且随包分发**（`rpa/config.json:4` + `build_signed_client.ps1:45`），泄露即可冒充设备触发真实发送。
- **[P1] `client.py` 是占位客户端会静默假回写**：`client.py:48-51` 领到任务不发消息却回写 `status:"dry_run"` 成功态，误运行即静默数据错误。应删除/隔离，被控端保留单一入口。
- **[P1] 点位靠硬编码比例 + DPI 换算**（`0.18`/`0.20`/`0.72` 等，单主屏假设），企微升级/改分辨率/多屏即失效——debug 截图里持续出现 `manual_scroll/fit_probe` 就是点位尚未收敛的现场证据。
- **[P1] 可观测性全靠 `print` 到 stdout**，无结构化日志、无 trace_id、无告警，现场无人值守时 MTTR 极高。
- **[P2] outbox 阻塞策略**：一条回写永久失败会永久阻塞该设备所有发送（`result_outbox.py:68-69`）。

**战略定位建议**：**投产后不应把 RPA 当稳定主通道承诺 SLA。** 消息读取（回读）应尽快交给已接入的会话存档 API——把「发送是否落地」的判定从「截图能不能读到我发的字」改为「存档流水里有没有这条 msgid」，可砍掉最不稳定的四级回读兜底与大量 OCR/ARK 成本。RPA 收敛到最小职责：仅「把已审核内容粘贴进目标会话并按发送」，并明确标注为「尽力而为、不承诺 SLA」的降级执行器。

### 2.6 运维部署与测试（运维 3.0 / 测试 6.0）

**核心问题：功能测试扎实、健康端点出彩，但生产承重墙（CI/CD、TLS、日志、监控、备份）几乎全缺。**

- **[P0] 无 TLS**：`docker-compose.yml:6-7` 直接暴露 `8000`，nginx conf 只 `listen 80`，管理登录与 PII 明文过公网。
- **[P0] 无 CI/CD**：无 `.github/`、无 pytest 配置，30+ 测试从不在提交时自动跑，部署靠手工 SFTP + 重建，无版本记录无回滚。
- **[P1] 无日志基础设施**：全 `app/` 无 `logging` 配置，compose 未限制 json-file 大小（会撑爆磁盘）；**无监控/告警/Sentry**：`/api/ops/health` 设计很好但是拉模式，critical 状态无人主动知晓；**容器 root 运行 + 无 healthcheck + 无资源限制 + 无优雅关闭**；**单机单点 + `latest` tag 无法回滚**。
- **[P2] 单 uvicorn worker**（多核用不上）；备份与主库同机同盘无异地。
- **测试**：关键路径覆盖好（`test_send_task_validation.py` 1903 行、数据隔离、AI 安全、限流、健康看板都有），但**全部进程内 SQLite 内存库，生产 PG 路径零测试**——这正是备份 bug 没被测出的原因；无覆盖率门禁、无 E2E、无部署冒烟。
- **正面确认**：客户端依赖（pywinauto/pywin32）已正确从 `requirements.server.txt` 剥离；依赖版本 pin 精确。

**投产前必补 Top 5**：① 修备份 + 真实「删库→恢复」演练；② 上 TLS（Caddy 自动证书最省事，8000 仅本地监听）；③ 建 CI（push 跑 pytest，合并触发部署）；④ 监控告警（Sentry + 外部探针轮询 health，critical 推企微/钉钉）；⑤ 集中日志 + 轮转。

### 2.7 产品与运营视角（补充）

技术之外，从「真正投入使用」角度需要同步就位的：

- **合规前置**：系统处理未成年人 + 家长敏感沟通数据，正式运营前应完成——① 个人信息处理告知—同意链路（尤其监护人同意）；② 数据处理者/受托处理协议；③ 会话存档 API 的授权边界与 RPA 自动化发言的法务/风控复核（RPA 审查中已标注为高敏感区）。建议投产前过一遍法务。
- **消息通道战略**：当前「RPA 发 + RPA 读」双脆弱。方向应是「存档 API 读 + RPA 收敛为最小发送执行器」，并规划官方发送能力（应用消息/群机器人）成熟后逐步迁移。这决定了产品能承诺给客户的稳定性。
- **可运营性缺口**：无使用数据埋点/运营看板（区别于技术健康看板）、无租户级配额与计费、无多机构隔离的成熟度验证（水平越权防护目前是 opt-in，重名 coach 会互看数据，`main.py:582`）。若要做多机构 SaaS，租户隔离需从「靠每端点主动调用作用域」升级为「默认强制作用域」。
- **AI 质量运营**：满意度评分等业务口径散落前端硬编码（`app.js:897-908`），前后端口径易不一致；AI 生成质量缺少线上评测闭环（虽有 `agent_eval` 服务，需确认是否接入线上）。
